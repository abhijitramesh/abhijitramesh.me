Problem

llama.cpp's WebGPU backend lacked systematic benchmarking across browsers, hardware, and quantization variants. There was no way to measure inference performance (token throughput, latency) or validate numerical correctness as the backend evolved. Testing required manual browser setup, lacked reproducibility across machines, and had no shared submission path so contributor data couldn't aggregate into a useful signal.

Contribution

Built webgpu-bench, an automated benchmarking framework that compiles llama.cpp to WebAssembly, runs inference in real browsers via WebGPU, and measures both performance and numerical correctness. Pivoted the leaderboard to a HuggingFace dataset as the single source of truth, with a HF Space static dashboard that rebuilds from the dataset on every push. Added a one-click in-browser Run page that lets any WebGPU user contribute results via HuggingFace OAuth with no local setup. Layered SkyPilot/RunPod automation so cloud GPUs can submit too.

Technical details

Compiles llama.cpp (git submodule) to two WASM variants using Emscripten: JSPI for Chrome's native async support, Asyncify for Safari. Playwright drives Chromium; WebDriverIO with a manual safaridriver lifecycle drives real Safari (native WebGPU only). Express server with HuggingFace caching proxy eliminates redundant downloads on local runs; the HF Space surface uses OPFS for in-browser model caching. Five exported WASM functions (init, load, run, eval_tokens, exit) provide benchmark control. Bench worker runs off the main thread with a HeapFS staging layer and a rolling cache to keep memory bounded, plus a buffered fallback for mobile browsers without transferable streams.

Validates numerical correctness via forced-decoding consistency measurement: CPU baseline generates reference token IDs, then GPU decoding is tested against the same context at each position, isolating quantization kernel issues from cascading divergence. Tracks 10 models with 194 quantization variants. Every benchmark record is stamped with the llama.cpp submodule commit hash so results stay attributable to a specific backend revision. Metrics: prefill/decode tokens-per-second, wall time, agreement rate vs CPU, browser version, platform/arch, and CPU baseline alongside the GPU run.

Outcome

Canonical deploy lives at the HuggingFace Space (abhijitramesh-webgpu-bench.static.hf.space, with custom domain webgpu-bench.abhijitramesh.me as alias). Dashboard merges the live HF dataset with submitter attribution, surfaces CPU vs WebGPU comparison views, the readable llama.cpp version on each row, error analysis grouped by OOM/timeout/abort, dark/light theme switcher, and shadcn-inspired UI redesign. The one-click Run page detects its surface (localhost / HF Space / other host) and adapts the model index, cache layer, and submission path accordingly. Crash isolation: Safari rewritten with a manual safaridriver lifecycle and hard-kill SIGKILL timeouts so a single browser hang doesn't take down the queue. Variant-first execution with model-at-a-time caching keeps disk usage stable, and --resume skips already-completed browser × variant × GPU-layer combos. 53 commits in 18 days of focused development culminated in the HF dataset pivot replacing the old PR-based submission flow.
