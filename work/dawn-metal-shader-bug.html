Problem

A WebGPU shader executing on Apple Metal via Dawn produced incorrect output when reading from a storage buffer. The shader mul_mat_reg_tile_q4_K_f32_exact.wgsl extracts bytes from three fixed u32 words inside a loop, expecting debug output [0xdeadbeef, 1, 2, 3] but observing zeros in indices 1–3. The failure was specific to Chrome on Apple Silicon; Safari passed. Only occurred with default Dawn settings (robustness enabled, integer division polyfills enabled).

Contribution

Created a public minimal-repro repository with two variants: a browser-based HTML demo and a native C++ runner using CMake, both isolating the shader bug and demonstrating the Metal/robustness interaction without external dependencies.

Technical details

The shader reads a constant buffer of three u32 values and extracts individual bytes in a loop. Two Dawn compiler toggles independently masked the bug: disable_robustness (memory safety checks) and disable_polyfills_on_integer_div_and_mod (integer operation instrumentation). This pattern suggests either robustness-pass interaction or a polyfill-induced register pressure issue on Metal.

Outcome

Bug report repo serves as a shared test case for Dawn maintainers. The isolation—both cross-platform (browser vs. native) and cross-browser (Chrome vs. Safari)—narrows the problem to Dawn's Metal backend and compiler flags rather than driver or shader logic. Two forks indicate other developers encountered the same issue.
